# Codebase Audit & Improvement Plan — 2026-04-26

Master plan for the testing / infrastructure / deployment audit. Each task has a
corresponding GitHub issue once this plan is approved. Work the tasks in the
listed order. Do not bundle tasks across PRs.

## Goal

Move the test, infrastructure, and deployment story from "fragile, with known
holes" to "guarded by CI, with one source of truth, with a recoverable
production." UI quality is out of scope. Domain correctness is out of scope.

## Workflow rules — apply to every task

1. **Worktree per task.** Use
   `git worktree add ../rivcomocktrial-<task>    -b <branch>` from the repo
   root. Never commit to `main`.
2. **Branch naming.** `feat/<topic>` for behavior changes, `fix/<topic>` for bug
   fixes, `docs/<topic>` for doc-only changes, `ci/<topic>` for workflow
   changes, `refactor/<topic>` for moves.
3. **TDD by default for behavior changes.** Write a failing test first, see it
   fail with a clear message, then make it pass. Skip only when the task is
   config or docs.
4. **No mocks.** All integration tests hit the real local PocketBase in the
   isolated test container (`docker-compose.test.yml`, port 28090). Use existing
   helpers in `web/src/lib/test-helpers/pb-admin.ts` and
   `tests/e2e/helpers/pb.ts`.
5. **Run before reporting done.** The user does not code in JavaScript and
   cannot easily verify a diff by reading. Every task ends with running the
   verification commands listed under "Verify" and pasting the output (or its
   summary) in the PR description.
6. **One PR per task.** No bundling unrelated work. If a task uncovers a
   separate bug, open a new issue rather than fixing in the same PR.
7. **Read-before-write at task start.** Re-read the files listed under "Files"
   before editing. State may have drifted since this plan was written.
8. **Patch-bump per PR, update CHANGELOG.md.** Per the project tagging policy.

### 6. Hash-pin PocketBase binary in Dockerfiles

**Background.** `backend/Dockerfile` downloads
`pocketbase_${PB_VERSION}_linux_${TARGETARCH}.zip` from GitHub releases without
checksum verification. A compromise of the GitHub release (or a MITM during
build) deploys an unverified binary to production on the next push. Standard
supply-chain hardening.

**Out of scope.** Do not bump the PocketBase version in this PR. Pin the version
that's already there.

**Files.** - `backend/Dockerfile` - `backend/Dockerfile.dev`

**Steps.**

1. Read `backend/Dockerfile` to find the current `ARG PB_VERSION` value.
2. Download both arch zips locally and compute SHA256:
   ```
   curl -sL https://github.com/pocketbase/pocketbase/releases/download/v0.36.3/pocketbase_0.36.3_linux_amd64.zip | sha256sum
   curl -sL https://github.com/pocketbase/pocketbase/releases/download/v0.36.3/pocketbase_0.36.3_linux_arm64.zip | sha256sum
   ```
3. Verify the values against the official `checksums.txt` published by
   PocketBase for that release. Fail loudly if they don't match.
4. Add to `backend/Dockerfile`:
   ```
   ARG PB_SHA256_AMD64=<value>
   ARG PB_SHA256_ARM64=<value>
   ```
   ```
   sh -c 'case "$TARGETARCH" in \
       amd64) echo "${PB_SHA256_AMD64}  pocketbase.zip" | sha256sum -c - ;; \
       arm64) echo "${PB_SHA256_ARM64}  pocketbase.zip" | sha256sum -c - ;; \
       *) echo "Unsupported arch: $TARGETARCH" && exit 1 ;; \
   esac'
   ```
5. Apply the same pattern to `backend/Dockerfile.dev`.
6. Build both Dockerfiles locally for the host arch only:
   `docker build -f backend/Dockerfile .` — confirm green. Repeat with
   `Dockerfile.dev`.
7. Sanity test: temporarily change one byte of the SHA in the Dockerfile,
   rebuild, confirm the build fails at the `sha256sum -c` step. Revert.

**Acceptance.**

- Both Dockerfiles fail to build if the downloaded binary doesn't match the
  pinned SHA256.
- A version bump requires updating both the version and the SHAs in the same
  change. Document this in a comment above the SHA ARGs:
  `# Update both ARGs together when bumping PB_VERSION.`

**Verify.**

- `docker build -f backend/Dockerfile .` succeeds.
- Tampered SHA test (above) fails as expected.
- Paste both build outputs (or relevant tail) in the PR.

### 7. PocketBase data backup policy (Fly volume snapshots)

**Background.** A single Fly volume holds `pb_data/data.db`. No visible backup

data loss. Fly provides free, automatic volume snapshots — they just need to be
enabled and documented.

**Out of scope.** Do not write a custom backup hook in PocketBase. Do not add S3
sync. Fly's built-in snapshots are enough for current scale and avoid new attack
surface.

**Files.** - New: `docs/backups.md` - `CLAUDE.md` (one-line pointer)

**Steps.**

1. Inspect current state for both apps:
   ```
   fly volumes list -a rivcomocktrial
   fly volumes list -a rivcomocktrial-staging
   ```
   ```
   fly volumes snapshots list <volume-id> -a <app>
   ```
2. If snapshot retention is less than 7 days, set it:
   ```
   fly volumes update <volume-id> --snapshot-retention 14 -a <app>
   ```
3. Take an explicit snapshot of production right now to anchor a known-good
   baseline:
   ```
   fly volumes snapshots create <volume-id> -a rivcomocktrial
   ```
4. Write `docs/backups.md` with three sections:
   - **What is backed up.** Fly volume snapshots are volume-level, taken daily,
     retained 14 days.
   - **What is NOT backed up.** Anything outside `pb_data` (hooks and migrations
     are in git; secrets are in fly secrets).
   - **Recovery procedure.** Step-by-step `fly volumes fork` command, then
     `fly machine update` to attach the forked volume, then verify with
     `fly logs`. Include rough time estimate ("expect \~5 minutes to recover
     from a snapshot").
5. Add to `CLAUDE.md` under a new "Operations" section: "Backups: Fly volume
   snapshots, 14-day retention. See `docs/backups.md` for recovery procedure."
6. **Test the recovery procedure once** in staging. Pick a non-production
   snapshot, fork it, attach, confirm data is intact, then detach and clean up.
   Document any deviations from the written procedure in `docs/backups.md`.

**Acceptance.**

- 14-day snapshot retention on both prod and staging volumes.
- Anchor snapshot taken on prod, dated.
- `docs/backups.md` exists with the three sections.
- Recovery procedure has been exercised once in staging.

**Verify.**

- `fly volumes snapshots list <prod-volume-id> -a rivcomocktrial` shows the
  anchor snapshot.

### 10. Pre-deploy `pb_data` snapshot in CI

**Background.** Task 7 enables daily Fly snapshots. That covers 24-hour
granularity. A bad migration deployed at 14:00 corrupts data after the daily
snapshot at 02:00 — recovery loses 12 hours of work. A pre-deploy snapshot
closes this window: every deploy gets its own snapshot, tagged with the commit
SHA.

**Out of scope.** Do not add post-deploy verification beyond the existing
migrate-check job. Do not add automatic rollback — that needs human judgment for
now.

**Files.** - `.github/workflows/deploy.yml`

**Steps.**

1. Re-read `deploy.yml`. Note the existing `migrate-check` job and how
   staging/production gates work.
2. In both the staging and production deploy jobs, add a step **before**
   `flyctl deploy`:
   ```
   - name: Snapshot pb_data before deploy
     run: |
       VOLUME_ID=$(flyctl volumes list -a $APP --json | jq -r '.[0].id')
       flyctl volumes snapshots create "$VOLUME_ID" -a $APP
     env:
       FLY_API_TOKEN: ${{ secrets.FLY_API_TOKEN }}
       APP: rivcomocktrial-staging  # or rivcomocktrial for prod
   ```
3. After `flyctl deploy` completes, log the snapshot ID in the job summary so
   it's easy to find for rollback.
4. Document in `docs/backups.md` (added in Task 7): "A snapshot is taken before
   every deploy. To roll back, fork the snapshot tagged just before the bad
   deploy."

**Acceptance.**

- Every deploy run produces a snapshot listed in `fly volumes snapshots list`.
- The snapshot is taken before any code change touches the live volume.

**Verify.**

- Trigger a staging deploy. Confirm a fresh snapshot exists.
- Paste the snapshot ID in the PR description.

**Risk / rollback.** If snapshot creation fails, deploy should NOT proceed (fail
the job). This protects data even if it briefly blocks a deploy.
